Dans votre cas, il n'y avait pas un seul problème, mais trois problèmes de réseau et de timing combinés :
1. Le problème du "Mauvais Numéro" (Conflit de Ports)
Dans Docker, chaque container a sa propre adresse.
Votre Keycloak était configuré pour écouter sur le port 9080 (c'est le standard JHipster 8).
Votre Application (Spring Boot) essayait d'appeler Keycloak sur le port 8080 par défaut.
Résultat : L'application "sonnait" à la mauvaise porte. Keycloak ne répondait pas, d'où l'erreur Connection refused.
2. Le problème de "l'Adresse" (Localhost vs Hôte)
C'est le piège n°1 de Docker :
Quand l'application est dans un container, pour elle, localhost désigne elle-même (son propre container).
Elle cherchait donc Keycloak à l'intérieur de son propre container au lieu de sortir sur le réseau Docker.
La solution a été de lui dire d'appeler http://keycloak:9080. Dans Docker, le nom du service sert d'adresse IP.
3. Le problème de "l'Impatience" (Le Timing)
Keycloak 26 est un serveur lourd basé sur Quarkus. Au démarrage :
Il doit d'abord attendre que PostgreSQL soit prêt.
Il doit ensuite créer toutes ses tables (plus de 100 tables).
Enfin, il génère ses clés de sécurité.
Le crash : L'application Spring Boot tentait de se connecter dès la première seconde. Keycloak était encore en train de "travailler" et refusait la connexion.
La solution : L'ajout de JHIPSTER_SLEEP=90 a forcé l'application à attendre que Keycloak ait fini son installation avant de tenter le premier contact.
4. Le problème du "Nom de Famille" (Configuration Spring)
Spring Boot est très strict sur les noms des variables d'environnement.
Dans votre fichier de configuration Java (application-prod.yml), vous aviez nommé votre fournisseur de sécurité keycloak.
Mais dans Docker, on essayait parfois de le configurer avec le nom oidc.
Résultat : Spring Boot ignorait vos réglages Docker et reprenait ses valeurs par défaut (qui étaient localhost:8080), ce qui nous ramenait aux problèmes 1 et 2.
En résumé :
Pour que cela marche, il fallait que tout le monde soit parfaitement aligné :
Appeler le bon nom (keycloak).
Appeler le bon port (9080).
Attendre le bon moment (JHIPSTER_SLEEP).
Utiliser les bons noms de variables (SPRING_SECURITY_..._KEYCLOAK_...).
Maintenant que cet "alignement des planètes" est fait, votre infrastructure est solide. Vous pouvez redémarrer vos containers autant de fois que vous voulez, le système saura se retrouver tout seul.

Problème de port 8082 déjà utilisé:
Dans ton fichier docker-compose.yml, tu as un service nommé esigmap_app qui expose le port 8082:8082.
Quand ton script fait docker compose up -d, Docker démarre l'application à l'intérieur d'un conteneur et prend le port 8082.
Juste après, ton script tente de lancer java -jar target/esigmap... directement sur ta machine hôte, qui essaie aussi de prendre le port 8082.
C'est le conflit : deux applications essaient d'utiliser le même port 8082 en même temps.

SOlution:

Option A : Tu veux développer et voir tes changements (Recommandé)
Dans ce cas, on utilise Docker uniquement pour la "base" (Postgres, Keycloak, Kafka, Minio) et on lance l'application avec le script pour pouvoir débugger facilement.
Modifie ton docker-compose.yml : Commente ou supprime la section esigmap_app.
Relance ton script.
Option B : Tu veux que tout tourne dans Docker
Dans ce cas, tu ne dois pas lancer java -jar dans ton script.
Supprime la ligne java -jar ... & dans ton script start.sh.
Laisse Docker gérer le lancement de l'app. (Mais attention, à chaque changement de code, tu devras reconstruire l'image Docker).
